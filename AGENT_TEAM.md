# Agent 团队定义

这个文件定义了 `AI Internship Agent` 项目的长期固定 Agent 团队。
这支团队是项目资产，不是一次性对话技巧。

## 1. 目的

保留这支团队，是为了让后续工作不必每次都重新设计协作结构。

默认规则：

- 主代理默认承担项目经理和最终整合者角色
- 各模块负责人只负责自己边界内的工作
- 跨模块决策统一上收给主代理

## 2. 团队编制

### Main agent / PM

负责：

- 任务拆解
- 模块分配
- 依赖协调
- 跨模块决策
- 最终验证
- 最终用户汇报

不可转交：

- 边界裁决
- 最终验收
- 冲突解决

### User lead

负责：

- 认证
- 登录 / token 流程
- 用户 CRUD
- 当前用户上下文
- 用户侧访问规则

### Resume lead

负责：

- 简历 CRUD
- 文件元数据
- 简历智能能力
- 简历服务层行为
- 简历 API 行为

### Job lead

负责：

- 岗位 CRUD
- 搜索 / 筛选逻辑
- 岗位侧业务规则
- 岗位 API 行为

### Interview lead

负责：

- 面试题库
- session / record 行为
- 面试智能能力
- 面试 API 行为

### Tracker lead

负责：

- 投递追踪生命周期
- 追踪状态流转
- 追踪查询
- tracker API 行为

### LLM/Core lead

负责：

- provider 适配器
- 工厂 / 配置解析
- 共享 agent 基础设施
- prompt / runtime 集成规则

### Data/Test lead

负责：

- 测试补充与回归安全
- schema / entity / repository 漂移检测
- 集成测试对齐
- 质量门支持

## 3. 边界规则

- 模块负责人只能修改自己模块内的实现细节
- 共享基础设施归 `LLM/Core lead` 统一负责
- 用户身份与授权语义归 `User lead` 牵头，最终裁决归 `Main agent`
- 跨模块所有权冲突只能由 `Main agent` 解决
- 任何模块负责人都不能私自为共享问题再造第二套实现路径

## 4. 任务路由规则

新任务到来时，默认按下面规则路由。

### 默认交付顺序

团队不再默认连续按业务线纵向切片推进。

从现在开始，阶段顺序固定为：

1. `Layer-first`
   - 共享 `data_access` 合同
   - 共享 `business_logic` 语义
   - 共享 `presentation` 路由 / 错误语义
   - 共享 `core` 运行时规则
2. `Domain-delivery`
   - 一次只推进一个有边界的业务域

这意味着：

- 主代理要先判断当前阶段是基础收敛阶段还是业务交付阶段
- `LLM/Core lead` 和 `Data/Test lead` 是基础阶段的默认参与者
- 业务负责人不能在领域阶段里私自发明新的持久化模式或 provider 模式

## 5. 标准汇报格式

所有子代理都必须按下面顺序回报：

1. 改动文件
2. 已实现能力
3. 剩余风险
4. 已运行测试
5. 测试结果
6. 下游依赖

主代理汇总时默认包含：

1. 当前阶段目标
2. 本阶段激活的角色
3. 各角色交付了什么
4. 主代理做了哪些跨模块决策
5. 已完成验证
6. 剩余风险
7. 推荐下一阶段

## 6. 全项目开发团队（端到端长期负责）

为满足“负责整个项目开发，而不是只负责某一部分”的要求，固定团队扩展为全生命周期协作编制。

### 6.1 端到端覆盖范围

该团队统一负责以下全链路事项：

- 产品方向与需求拆解
- 后端架构与领域能力迭代
- 前端产品化与体验一致性
- AI 能力接入（preview / persist / history）
- 数据库迁移与版本链路稳定
- 测试体系与回归门禁
- Demo 与发布资产维护
- 文档、记忆、计划持续更新

### 6.2 全项目常驻角色

在原固定编制基础上，增加以下常驻角色：

- `Frontend lead`
  - 负责前端应用壳、页面路由、状态管理、API 对接、UI 一致性
- `Design/Product lead`
  - 负责产品信息架构、交互流、Figma 对齐、文案与可演示性
- `DevOps/Release lead`
  - 负责运行脚本、Compose 路径、部署文档、发布检查清单

> 说明：`Main agent / PM` 仍是唯一总协调与最终验收角色。

### 6.3 全项目阶段责任矩阵

- 需求与计划阶段：
  - 主责：`Main agent / PM` + `Design/Product lead`
- 架构与基础收敛阶段：
  - 主责：`LLM/Core lead` + `Data/Test lead` + `DevOps/Release lead`
- 领域能力开发阶段：
  - 主责：`User/Resume/Job/Interview/Tracker lead`
- 产品体验与前后端联调阶段：
  - 主责：`Frontend lead` + 各业务 lead
- 回归、验收与发布阶段：
  - 主责：`Data/Test lead` + `DevOps/Release lead` + `Main agent / PM`

### 6.4 启动与交付规则

每次新阶段默认执行：

1. `Main agent / PM` 宣布阶段目标与非目标
2. 按责任矩阵激活本阶段所需角色（可多角色并行）
3. 每个角色按统一模板回报：
   - 改动文件
   - 已实现能力
   - 剩余风险
   - 已运行测试
   - 测试结果
   - 下游依赖
4. `Main agent / PM` 统一整合并给出“下一推荐任务”

### 6.5 统一成功标准

全项目团队的阶段完成必须同时满足：

- 架构边界未破坏（层次依赖正确）
- 相关测试集可运行并通过
- 前后端契约一致
- 可演示链路可跑通
- 记忆与计划文件已同步更新
