# 作品集后端概览

这份文档面向面试官、评审者和作品集阅读者。

## 这个后端展示了什么

这个项目展示的是一个具备以下特征的后端：

- 分层清晰
- 支持 AI 能力
- 用户隔离明确
- 有测试支撑
- 可通过 Docker Compose 部署和演示

这个仓库被有意识地定位为一个**后端作品集交付物**，而不是一个完整生产平台。

## 核心能力矩阵

| 领域 | 预览 / 生成能力 | 已持久化结果 | 历史 / 查询 |
|---|---|---|---|
| Resume | 摘要、优化建议 | 优化建议、摘要 | 支持 |
| Interview | 题目生成、答案评估 | record 评估结果 | 通过 records 支持 |
| Job | 匹配预览 | 匹配结果 | 支持 |
| Tracker | 建议预览 | 建议结果 | 支持 |

## 架构形态

这个后端遵循三层结构：

- `presentation`
- `business_logic`
- `data_access`

共享 AI / 运行时基础设施位于 `core`。

这意味着这个项目并不只是一些 API 文件的拼接，而是被组织成一个能够展示下列工程能力的后端：

- 关注点分离
- 可复用的服务合同
- 仓储层持久化边界
- 明确的运行时 / 配置处理

## 官方演示路径

当前这个作品集版本的官方演示部署路径是：

- Docker Compose

选择这条路径的原因是：

- 更容易让评审者复现
- 更适合面试场景中的快速演示
- 足以体现部署意识，但不会过度夸大基础设施成熟度

## 明确纳入当前交付范围的内容

- 有边界的 AI 辅助后端流程
- 已认证的用户主路径
- 受保护的领域 API
- 部分 AI 结果持久化
- 已文档化的迁移与健康 / 就绪检查行为
- 具备发布风格的 Compose 路径

## 明确不在当前交付范围内的内容

- Kubernetes 就绪部署
- 完整生产级认证 / session 生命周期
- 企业级后台任务处理与作业编排
- 前端应用交付

## 为什么这些边界很重要

这个项目最适合被表述为：

> 一个结构清晰、有测试支撑、带 AI 能力、并且可通过 Compose 复现的后端作品集项目

而不适合被表述为：

> 一个已经把部署和运维问题全部解决的完整生产平台

把边界说清楚，会让项目显得更成熟，而不是更弱，因为取舍是明确的。
