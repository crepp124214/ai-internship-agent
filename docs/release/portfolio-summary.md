# Portfolio Summary - AI 实习求职 Agent System

## 背景问题
求职学生在实习申请过程中面临四个主要痛点：
1. 简历不知道怎么写 — 不知道 HR 看重什么经历，不知道如何针对不同岗位调整简历
2. 面试不知道考什么 — 准备方向不明确，刷题效率低，模拟面试资源有限
3. 岗位信息分散 — 需要在多个平台搜索，信息整合耗时耗力
4. 缺乏反馈机制 — 不知道自己简历写得怎么样，面试回答得好不好

这些问题导致学生求职准备缺乏针对性，效率低下，成功率不高。

## 解决方案
AI 实习求职 Agent 系统通过 AI 技术提供四大核心功能，形成闭环求职准备流程：
- **简历定制优化**：上传简历，根据目标岗位（JD）生成针对性的优化建议，包括关键词提取、技能匹配、经历描述改进
- **AI 模拟面试**：基于简历和目标岗位，生成个性化面试题库。支持多轮对话式练习，实时评分、改进建议、思路引导
- **岗位智能匹配**：分析简历与岗位的匹配度，量化打分，指出简历中与岗位要求不匹配的地方
- **Agent 助手**：自然语言交互的智能助手，可搜索公司招聘官网、分析 JD、回答求职相关问题

## 核心功能
### 1. 简历定制
- 上传简历文件（支持 txt、md、json 格式）
- 根据目标岗位自动生成优化建议
- 支持 JD 定向优化：基于具体岗位重写简历内容
- 提供简历摘要生成和多版本管理
- 包含匹配报告：关键词覆盖率、匹配分数、gap 分析

### 2. AI 模拟面试
- 多轮对话式面试练习
- 实时评分和即时反馈
- 支持题目生成和题库管理
- 面试教练模式：专业维度评分（专业知识、思路清晰度、表达能力等）
- 生成详细的面试复盘报告

### 3. 岗位匹配
- 简历与岗位的多维度匹配度分析
- 量化匹配分数展示
- 具体指出匹配gap和改进建议
- 支持岗位导入/创建和收藏功能
- 一键流转：从岗位直接跳转到简历优化

### 4. Agent 助手
- 自然语言交互界面
- 联网搜索公司招聘官网
- JD 要求分析和解读
- 求职相关问题智能回答
- 可复用的助手面板组件，嵌入简历和岗位页面

## 架构亮点
### 1. Agent 运行时架构
- 基于 ReAct (Reasoning + Acting) 模式的统一 Agent 执行框架
- 包含 StateMachine、LLM Adapter、MemoryStore、ContextBuilder、ToolRegistry + BaseTool、AgentExecutor
- 支持工具注册、状态管理、记忆存储
- 实现了 Agent 的标准化生命周期管理

### 2. 多 Agent 协作
- 简历 Agent、岗位 Agent、面试 Agent 三个专业 Agent 独立运行
- 各自专注垂直场景，互不干扰
- 通过统一的 Agent Runtime 协作
- 实现了专业化分工和协同作战

### 3. LLM 灵活配置
- 支持多个大模型提供商：OpenAI、MiniMax、DeepSeek、智谱 AI 等
- 按 Agent 单独配置不同的 provider 和 model
- 真正实现 BYOK (Bring Your Own Key) 功能
- 包含 API Key 安全存储和 Base URL 配置
- Temperature 参数可调节 AI 创造性和确定性平衡

### 4. 前后端分离
- 前端：React 18 + TypeScript + Vite + TanStack Query + Zustand
- 后端：FastAPI + SQLAlchemy 2.0 + PostgreSQL + Redis
- 架构清晰易于扩展和维护
- 通过 RESTful API 进行前后端通信
- 采用 JWT token 进行身份认证

## 我的关键工程判断
1. **砍掉 Tracker，专注主链路**：早期版本包含求职申请追踪功能，但经评估认为这是非核心功能，砍掉后能更专注于简历优化→面试准备的核心闭环
2. **Agent Runtime 统一抽象**：与其为每个功能重新实现 Agent 逻辑，不如抽象出统一的运行时框架，提高开发效率和一致性
3. **暖色调设计系统**：采用珊瑚#e85d4c + 薰衣草#8b7cf8 的配色方案，区别于常见的冷色调科技风，营造温暖亲切的求职助手形象
4. **双栏布局 + 浮动工具栏**：在简历优化页面采用左右双栏布局（编辑区 + AI 功能区），以及选中文本时出现的浮动工具栏，大幅提升交互效率
5. **SSE 流式对话**：采用 Server-Sent Events 技术实现实时流式对话，比轮询方式更省资源，用户体验更流畅

## 已知边界与后续方向
### 已知边界
1. **演示环境 vs 生产环境**：Docker 部署使用 mock LLM，提供即用演示体验；生产环境需要用户配置真实 API Key 才能获得完整 AI 能力
2. LLM 配置门槛：需要用户自行配置有效的 API Key 才能获得最佳 AI 体验
3. 岗位数据来源：目前主要依赖用户导入或手动创建岗位，缺少自动化的外部岗位聚合
4. 简历解析限制：对复杂格式的 PDF 或 Word 简历支持有限，主要依赖纯文本格式

### 后续方向
1. 真实用户数据积累：通过真实用户反馈持续优化 AI 模型和算法
2. 外部数据源集成：集成公开的招聘网站 API 或进行网页爬虫以丰富岗位信息
3. 高级简历解析：增强对 PDF、Word 等格式的解析能力，支持更多简历格式
4. 移动端适配：考虑开发移动端应用或小程序，扩展使用场景
5. 数据可视化：增加求职进度可视化仪表盘，帮助用户更直观地看到自己的提升
6. 社区功能：探索用户之间的经验分享和互助功能的可能性

- 提高简历针对性和通过率
- 增加面试准备的效率和信心

- 最终实现求职成功率的显著提升

系统不仅是一个工具，更是一个全程陪伴的智能求职助手，让求职过程变得更有方向、更有效果、更有信心。
