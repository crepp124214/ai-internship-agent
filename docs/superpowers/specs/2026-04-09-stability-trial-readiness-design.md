# 稳定试用就绪设计方案（E2E）

日期：2026-04-09  
状态：Draft（待评审）

## 1. 目标与成功标准

本方案目标是将当前系统从“可演示”提升为“可稳定试用”，优先保障主链路可用性与失败可恢复性，不扩展新业务功能。

成功标准：

- 连续 20 次完整主链路（登录 -> 岗位探索 -> 简历优化 -> 面试准备 -> 设置中心）通过率 >= 95%
- P0/P1 问题为 0
- 所有失败均可定位并包含明确错误语义（可读提示 + request_id）

## 2. 范围与非目标

范围：

- 后端就绪门控与关键接口错误语义统一
- 前端全局服务状态检测与关键交互降级
- 前端路由完整性检查与修复（包含设置中心主页可访问）
- API + 前端主链路回归与稳定性连续跑

非目标：

- 不新增业务能力
- 不做大范围重构或跨模块 UI 重设计
- 不调整领域模型与核心业务流程

## 3. 方案总览

采用“三层防线”保证稳定试用：

1. 启动前防线：依赖与配置就绪门控，禁止“健康绿但不可用”
2. 请求时防线：统一错误翻译，关键接口不再随机 500
3. 前端体验防线：未就绪/降级状态可见，关键动作可禁用且可重试

## 4. 架构与组件设计

### 4.1 ReadinessGate（后端门控）

职责：

- 聚合 DB、Redis、关键配置状态
- 输出统一状态：`ready/not_ready` + `reason`
- 供 `/ready` 与关键接口前置检查复用

设计要求：

- 不改变现有分层方向：`presentation -> business_logic -> data_access`
- 检查逻辑集中在一个可复用模块，避免各路由重复探测

### 4.2 ApiErrorTranslator（后端错误翻译层）

职责：

- 统一将底层异常映射为稳定 HTTP 语义和错误码
- 保证关键链路错误结构一致：`code/message/retryable/request_id`

关键映射建议：

- 依赖不可用（DB/Redis）-> `503`
- 参数与状态冲突 -> `4xx`
- 未知错误 -> `500`（保留 request_id，隐藏内部细节）

错误码契约（最小集合）：

- `SERVICE_NOT_READY`：依赖未就绪，`retryable=true`
- `AUTH_REQUIRED`：未认证，`retryable=false`
- `INVALID_INPUT`：参数无效，`retryable=false`
- `RESOURCE_NOT_FOUND`：资源不存在，`retryable=false`
- `STATE_CONFLICT`：状态冲突，`retryable=false`
- `INTERNAL_ERROR`：未知错误，`retryable=true`

### 4.3 FrontendServiceStatus（前端服务状态源）

职责：

- 启动后探测 `/ready`
- 全局维护服务可用状态，供页面与按钮使用

行为：

- 服务未就绪：显示明确提示，禁止关键提交动作
- 服务恢复：自动解除禁用并提示可重试
- 启动时同时校验 API 端点连通性（host/port/baseURL），配置异常时给出显式提示

### 4.4 UserFacingFallbackPolicy（前端降级策略）

职责：

- 定义关键动作失败时的统一用户反馈与恢复路径

覆盖动作：

- 登录
- 登录状态探测与未登录引导（含 ProtectedRoute 恢复校验）
- 岗位收藏
- 简历优化提交
- 面试开始/提交回答/结束
- 报告查询

输出形式：

- 全局提示（服务不可用）
- 就地错误（表单或动作级）
- 明确重试入口

## 5. 数据流与错误流

主链路数据流：

1. 前端先读取 `FrontendServiceStatus`
2. 就绪则发起业务请求
3. 后端业务层异常进入 `ApiErrorTranslator`
4. 前端按错误 `code` 执行对应降级策略

错误流约束：

- 前端不得依赖字符串匹配判断错误类型
- 统一按 `code/retryable` 做交互分支
- 每次错误必须记录并透传 `request_id`

## 6. 测试与验收设计

### 6.1 API 稳定性回归

覆盖：

- 登录、岗位保存、简历优化、面试 start/answer/end、报告查询
- 依赖失效时返回一致 `503` 与错误码

### 6.2 前端主链路回归

剧本：

- 登录 -> 岗位探索 -> 简历优化 -> 面试准备 -> 设置中心

验证点：

- 按钮可用/禁用状态
- 错误提示可读性
- 重试入口与恢复路径
- 设置中心页面可访问（路由完整，无空页面/无响应跳转）

### 6.3 稳定性连续跑

执行：

- 连续 20 次完整主链路

判定：

- 通过率 >= 95%
- P0/P1 = 0
- 失败可归因（含 request_id）

失败定义：

- 任一关键 API 返回非 2xx 视为失败
- 页面白屏、关键按钮无响应、主流程中断视为失败

request_id 约束：

- 后端在关键错误响应中返回 `X-Request-ID`（或响应体同名字段）
- 前端将 request_id 展示到错误详情，供排障追踪

## 7. 风险与对策

风险 1：依赖波动导致间歇性 503  
对策：将依赖可用性前移到门控层，并在前端做可恢复提示。

风险 2：错误语义不一致导致前端处理分叉  
对策：统一错误结构并建立错误码契约。

风险 3：回归用例不足导致“看似稳定”  
对策：强制连续跑指标作为封板门槛。

风险 4：认证层临时绕过导致安全与验收失真  
对策：将 ProtectedRoute 恢复为必检项，并纳入主链路验收前置条件。

风险 5：路由缺失导致主链路天然断裂  
对策：在 E2E 前执行路由完整性检查，确保设置中心等关键入口可达。

## 8. 里程碑与退出条件

里程碑 M1：ReadinessGate + ApiErrorTranslator 接入关键链路  
退出条件：关键接口在依赖失效场景不出现裸 500。

里程碑 M2：FrontendServiceStatus + 降级策略完成  
退出条件：用户可见明确错误与重试路径。

里程碑 M3：回归与连续跑达标  
退出条件：20 次主链路 >=95%，无 P0/P1。

## 9. 与现有文档一致性

本设计遵循当前 `Release Candidate` 口径：

- 当前可“有条件交付”
- 覆盖率门禁以 `pytest.ini` 当前配置为准
- 本轮重点是稳定性与体验保障，不扩展功能范围

## 10. 已知阻断清单（执行前）

- 设置中心主页路由需确认可访问
- ProtectedRoute 不得处于临时绕过状态
- 前端 API 端口/baseURL 配置必须与后端运行端口一致
