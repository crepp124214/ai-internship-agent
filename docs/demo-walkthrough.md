# 前后端作品集演示脚本

这份文档用于把当前仓库演示成一套**真实可运行的 AI 求职产品原型**，而不是一组分散的后端接口。

## 演示目标

这次演示要证明 4 件事：

- 项目可以本地快速启动
- 前后端都能真实跑起来
- 登录与受保护页面链路可用
- `简历 / 岗位 / 面试 / 投递追踪` 可以串成一条完整产品路径

## 演示前准备

### 1）初始化环境

```bash
cp .env.example .env
cp frontend/.env.example frontend/.env
python scripts/migrate.py
python scripts/seed_demo.py
```

### 2）启动后端

```bash
make run
```

确认以下接口可访问：

- <http://127.0.0.1:8000/health>
- <http://127.0.0.1:8000/ready>

### 3）启动前端

```bash
cd frontend
npm run dev
```

默认地址通常是：

- `http://127.0.0.1:4173`

如果端口被占用，Vite 会自动切换到下一个可用端口（例如 `4174`）。

### 4）演示账号

- 用户名：`demo`
- 密码：`demo123`

## 正式演示顺序

### 1. 登录

打开前端登录页，使用演示账号直接登录。

讲解重点：

- 前端不是静态壳，已接通 Bearer 令牌鉴权
- 登录成功后会调用 `/api/v1/users/me`
- 未登录访问受保护页面会回退到登录页

### 2. 仪表盘

登录后进入仪表盘，说明其定位：

- 仪表盘不依赖新聚合接口
- 只是组合已有 `简历 / 岗位 / 面试 / 投递追踪` 结果

讲解重点：

- “最新 AI 输出”展示的是已存在的保存结果
- “快速开始”对应 4 条真实业务页面
- 首页是产品工作台，不是伪造统计面板

### 3. 简历（Resume）

进入简历页面，展示：

- 简历内容
- 摘要预览 / 保存
- 优化预览 / 保存
- 摘要历史
- 优化历史

讲解重点：

- 简历结果不是一次性返回，而是可留历史
- 历史结果会反映到仪表盘摘要区

### 4. 岗位（Jobs）

进入岗位页面，展示：

- 岗位详情
- 选择简历进行匹配
- 匹配预览
- 保存匹配
- 匹配历史

讲解重点：

- 岗位匹配依赖已有简历内容
- 保存后的匹配结果会回流到仪表盘

### 5. 面试（Interview）

进入面试页面，按顺序演示：

1. 生成题目
2. 选择一题保存到题库
3. 输入回答
4. 预览评估
5. 创建记录
6. 保存记录评估

讲解重点：

- 题目生成和记录保存是两步，不是伪造一键
- 当前前端已补齐“生成题目 -> 保存题目 -> 创建记录”的真实桥接

### 6. 投递追踪（Tracker）

进入投递追踪页面，展示：

- 投递列表或创建投递
- 建议预览
- 保存建议
- 建议历史

讲解重点：

- 投递建议依赖已有岗位 + 简历 + 投递上下文
- 保存后同样可形成历史记录

### 7. 回到仪表盘

最后返回仪表盘，解释：

- 首页如何组合四条业务线最新结果
- 为什么这是一套真实产品壳，而不是几张静态页面

## 一键验证步骤

若只想快速确认“当前可演示链路是否可用”，按下面顺序执行：

1. `python scripts/migrate.py`
2. `python scripts/seed_demo.py`
3. `make run`
4. `cd frontend && npm run dev`
5. 打开前端登录页
6. 使用 `demo / demo123` 登录
7. 进入 `/dashboard`
8. 依次打开 `/resume`、`/jobs`、`/interview`、`/tracker`

## 收尾话术（可直接复述）

> 这个项目现在已经不只是后端接口集合，而是一套可本地启动、可真实登录、可完成前后端联调演示的 AI 求职产品原型。它有清晰的三层架构、可追踪的 AI 结果历史，以及一套能承接这些能力的前端工作台。

## 当前边界

建议主动说明边界：

- 当前官方演示部署路径是 Docker Compose
- 不是 Kubernetes 就绪系统
- 没有完整刷新令牌（refresh token）/ 后台任务（worker）编排体系
- 当前重点是作品集级产品演示，不是假装生产级完工
