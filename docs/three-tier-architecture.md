# 三层架构设计文档

## 1. 架构概述

AI实习求职Agent系统采用经典的三层架构设计，将系统分为表现层、控制逻辑层和数据持久层，各层职责清晰，耦合度低，便于维护和扩展。

## 3. 各层职责详细说明

### 3.1 表现层 (Presentation Layer)

**位置**: `src/presentation/`

**职责**：
- 接受用户请求并格式化响应
- 数据验证与转换
- 错误处理与异常传播
- API接口定义
- 协议转换

**主要模块**：
- `api/v1/`: REST API接口
- `schemas/`: Pydantic数据验证模型
- `deps.py`: 依赖注入

**代码示例**：
```python
# src/presentation/api/v1/resume.py
@router.post("/")
async def create_resume(resume_data: ResumeCreate, db: Session = Depends(get_db)):
    """创建简历"""
    try:
        resume = await resume_service.create_resume(db, resume_data)
        return resume
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

### 3.2 控制逻辑层 (Business Logic Layer)

**位置**: `src/business_logic/`

**职责**：
- 核心业务逻辑实现
- 任务编排与调度
- 业务规则管理
- Agent协作
- 数据转换

**主要模块**：
- `agents/`: 专业化Agent实现
- `services/`: 业务服务（包含核心业务逻辑）

**代码示例**：
```python
# src/business_logic/services/resume_service.py
class ResumeService:
    async def create_resume(self, db: Session, resume_data: ResumeCreate):
        """创建简历业务逻辑"""
        # 业务规则验证
        if not self._validate_resume_data(resume_data):
            raise ValueError("无效的简历数据")

        # 调用数据访问层
        resume = resume_repository.create(db, resume_data)

        # 触发业务流程
        await self._trigger_resume_analysis(resume)

        return resume
```

### 3.3 数据持久层 (Data Access Layer)

**位置**: `src/data_access/`

**职责**：
- 数据存储与检索
- 数据库连接管理
- 数据转换
- 事务管理

**主要模块**：
- `entities/`: ORM实体模型
- `repositories/`: 数据访问对象
- `database.py`: 数据库连接与初始化

**代码示例**：
```python
# src/data_access/repositories/resume_repository.py
def create(self, db: Session, data: dict) -> Resume:
    """创建简历记录"""
    instance = Resume(**data)
    db.add(instance)
    db.commit()
    db.refresh(instance)
    return instance
```

### 3.4 核心基础设施层 (Core Layer)

**位置**: `src/core/`

**职责**：
- 系统配置管理
- 日志记录
- 异常处理
- 工具函数
- 第三方服务集成

**主要模块**：
- `agent/`: Agent基类与框架
- `llm/`: LLM接口抽象
- `memory/`: 记忆管理
- `tools/`: 通用工具库

## 4. 架构设计原则

### 4.1 单一职责原则 (SRP)

每个类、模块或函数应该有且仅有一个改变的理由。

### 4.2 开闭原则 (OCP)

软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。

### 4.3 依赖倒置原则 (DIP)

高层模块不应该依赖低层模块，两者都应该依赖抽象。抽象不应该依赖细节，细节应该依赖抽象。

### 4.4 接口隔离原则 (ISP)

客户端不应该被迫依赖于它不使用的方法。

### 4.5 迪米特法则 (LOD)

一个对象应该对其他对象有最少的了解。

## 5. 层间通信规则

### 5.1 依赖方向

```
表现层 → 控制逻辑层 → 数据持久层
↑                     ↑
└───────── 核心层 ─────┘
```

### 5.2 禁止跨层访问

- **表现层** 不应直接访问 **数据持久层**
- **控制逻辑层** 不应直接访问基础设施（应通过核心层）
- **数据持久层** 不应访问 **表现层** 或 **控制逻辑层**

### 5.3 依赖注入

通过依赖注入（DI）模式来管理依赖关系，提高模块的可测试性和可维护性。

## 6. 技术实现

### 6.1 框架与库

| 层次 | 技术选型 |
|------|----------|
| **表现层** | FastAPI, Pydantic |
| **控制逻辑层** | Python异步编程, Celery |
| **数据持久层** | SQLAlchemy 2.0, PostgreSQL, Redis, ChromaDB |
| **核心层** | Python标准库, 自定义工具函数 |

### 6.2 数据传输对象 (DTO)

使用Pydantic模型作为数据传输对象，在各层之间传输数据。

### 6.3 事务管理

在控制逻辑层或数据持久层实现事务管理，确保数据一致性。

## 7. 测试策略

### 7.1 单元测试

| 层次 | 测试重点 |
|------|----------|
| **表现层** | API接口测试, 数据验证 |
| **控制逻辑层** | 业务规则测试, 流程验证 |
| **数据持久层** | 数据操作测试, 事务管理 |
| **核心层** | 工具函数测试, 配置管理 |

### 7.2 集成测试

测试各层之间的交互和协作。

### 7.3 端到端测试

测试完整的业务流程，从用户请求到数据存储。

## 8. 错误处理

### 8.1 异常分层

| 层次 | 异常类型 |
|------|----------|
| **数据持久层** | 数据库异常, 数据访问异常 |
| **控制逻辑层** | 业务异常, 验证异常 |
| **表现层** | API异常, 客户端异常 |

### 8.2 错误传播

异常应该在适当的层次处理，如果无法处理，则向上传播到更高层次。

## 9. 性能优化建议

### 9.1 缓存策略

在表现层和控制逻辑层实现缓存，减少对数据持久层的访问。

### 9.2 异步处理

对长时间运行的操作使用异步处理，提高系统响应速度。

### 9.3 数据库优化

- 查询优化
- 索引设计
- 连接池管理

## 10. 部署架构

采用微服务或单体应用架构，通过Docker容器化部署。

### 10.1 开发环境

使用Docker Compose进行本地开发和测试。

### 10.2 生产环境

使用Kubernetes进行容器编排和管理。

## 11. 版本控制

使用Git进行代码版本控制，遵循语义化版本控制规范。
